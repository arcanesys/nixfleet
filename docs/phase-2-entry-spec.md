# Phase 2 entry spec

Pins the first cross-stream wire-up test: the minimum test that proves Stream A, Stream B, and Stream C hang together in production-shape for one signed artifact. Everything larger (magic rollback, compliance gate, freshness refusal) is Checkpoint 2 work that sits on top of this.

Cross-references: `docs/KICKOFF.md` §1 Phase 2, `docs/CONTRACTS.md` §I/§II/§III, `docs/trust-root-flow.md`, `ARCHITECTURE.md` §6 Phase 4.

Status: **proposed** — review + adopt as the spec for the Phase 2 implementation PR.

## 1. Goal

**Prove that a single CI-signed `fleet.resolved.json` round-trips through the whole stream stack without any bytes being reinterpreted:**

1. Stream B evaluates `mkFleet` → raw JSON.
2. Stream A's CI stamps `meta.{signedAt, ciCommit, signatureAlgorithm}` via `withSignature`.
3. Stream C's `nixfleet-canonicalize` produces the canonical bytes.
4. Stream A's `tpm-sign` produces a 64-byte signature over those exact bytes.
5. Stream C's `verify_artifact` accepts the artifact against the declared trust root.
6. One harness agent receives the verified artifact from a harness CP and logs it.

If any step reinterprets bytes (different canonicalization, different serde posture, different algorithm tag), the signature fails at step 5 and the harness scenario fails.

**Not proving (yet):** rollout actions, agent activation, probe execution, rollback, freshness refusal, multi-host coordination. Those are Checkpoint 2 scenarios.

## 2. Minimum viable scope

The test is a single harness scenario — `fleet-harness-signed-roundtrip` — that extends the smoke scaffold from #5:

| Piece | Source | State |
|---|---|---|
| `fleet.resolved.json` with CI-style metadata | fixture, baked at build time | generated by a small derivation from `mkFleet` on an ephemeral test input |
| `fleet.resolved.json.sig` | fixture, baked at build time | signed with an ephemeral ed25519 keypair generated in the same derivation |
| Trust config | declared in the scenario's host-VM module | single `ciReleaseKey.current = { algorithm = "ed25519"; public = <test-pubkey>; }` |
| CP stub | harness host-VM systemd service | serves the fixture + sig over mTLS (socat stays — no real CP binary needed yet) |
| Agent | harness microVM | fetches artifact + sig, calls `verify_artifact` via a thin CLI wrapper, logs on success |

Using ed25519 (not ecdsa-p256) for the first wire-up for a reason: it's the default algorithm, it's software-generateable inside a Nix derivation (no TPM dependency), and it keeps the harness reproducible on any host. The p256 path is exercised by Stream C's unit tests; the harness doesn't need to exercise it again until a Checkpoint 2 scenario explicitly tests cross-algorithm rotation.

## 3. Test sequence (per scenario run)

```
Build time (Nix derivation):
  1. Generate ephemeral ed25519 keypair — test-sign-key, test-verify-key.
  2. Call mkFleet on a hand-written small input → raw_fleet.json.
  3. Pipe raw_fleet.json through withSignature { signedAt = "<fixed ts>"; ciCommit = "<fixed sha>"; } → stamped.json.
  4. Pipe stamped.json through nixfleet-canonicalize → canonical.json.
  5. Sign canonical.json with test-sign-key → canonical.json.sig.
  6. Emit (canonical.json, canonical.json.sig, test-verify-key) to the scenario.

Harness time (VM):
  7. Host VM boots. CP stub serves canonical.json + canonical.json.sig on :8443 mTLS.
  8. Host VM writes /etc/nixfleet/test-trust.json with { algorithm = "ed25519"; public = test-verify-key; }.
  9. One agent microVM boots. Its curl+verify wrapper:
     a. Fetches canonical.json and canonical.json.sig from CP (mTLS).
     b. Reads /etc/nixfleet/test-trust.json.
     c. Calls `nixfleet-verify-artifact` CLI (new thin wrapper, see §6) with the four inputs.
     d. On success, logs `harness-roundtrip-ok: schemaVersion=<n> hosts=<n>`.

Assertion:
  10. testScript waits for the agent journal line within 120s. Absence or any verify error fails the scenario.
```

## 5. Success criteria (exit conditions)

The scenario PASSES iff:

- `harness-roundtrip-ok:` appears in the agent journal within 120s of `start_all()`.
- No `VerifyError` variant appears in the agent journal.
- The logged `schemaVersion` matches `1`.

The scenario FAILS (red signal, not a bug) iff:

- Any of `nixfleet-canonicalize` produces bytes that differ from what `withSignature`+`tpm-sign`/ed25519-sign consumed (byte-drift across streams).
- `verify_artifact` returns any `VerifyError` variant (signature mismatch, stale, unsupported algorithm, bad pubkey).
- mTLS handshake fails (cert shape drift).
- The fixture derivation reproduces non-identical output between two runs (signals non-determinism in `mkFleet` or canonicalize).

The scenario is WRONG (bug in the test itself, not the streams) iff:

- The ephemeral keypair changes between scenario eval and runtime.
- The timestamp in the fixture drifts (not frozen).
- The CP stub socat somehow reorders bytes.

## 6. `nixfleet-verify-artifact` — new thin CLI

New binary in `crates/nixfleet-verify-artifact/`. Exists purely so the harness agent can call `verify_artifact` from a shell-friendly entry point before Stream C's v0.2 agent skeleton lands with the same behavior baked in.

Shape:

```
nixfleet-verify-artifact \
  --artifact <path-to-canonical.json> \
  --signature <path-to-sig> \
  --trust-file <path-to-trust.json> \
  --now "<RFC 3339 timestamp>" \
  --freshness-window-secs <n>
```

Exit codes: 0 = verified, 1 = verify error (stderr carries the `VerifyError` variant + detail), 2 = argument error.

Implementation: ~50 LOC wrapping `nixfleet_reconciler::verify_artifact`. Adds no new proto types — consumes the existing ones.

Temporary lifetime: the binary is deleted the moment Stream C's v0.2 agent skeleton ships with an equivalent internal call site. It's harness-only plumbing.

## 7. Signed-fixture derivation

New Nix derivation (lives in `tests/harness/fixtures/signed/default.nix` or similar). Takes:

- `mkFleet` input (a small hand-written fleet declaration).
- `nixfleet-canonicalize` package.
- `pkgs.openssl` (for `genpkey` + `pkeyutl -sign`) or Rust `ed25519-dalek` via a small helper.

Produces:

- `canonical.json` — the JCS-canonical signed artifact.
- `canonical.json.sig` — the 64-byte ed25519 signature.
- `verify-pubkey.b64` — the base64-encoded verify key to embed in `test-trust.json`.

Determinism: all inputs are pure (no timestamps from the build host; the fixture's `signedAt` is a hand-set string like `"2026-05-01T00:00:00Z"`; the keypair is derived from a fixed test seed so rebuilds produce identical bytes).

```nix
  harnessLib.mkFleetScenario {
    name = "fleet-harness-signed-roundtrip";
    inherit cpHostModule;
    agents = { agent-01 = agent; };
    timeout = 600;
    testScript = ''
      start_all()
      host.wait_for_unit("multi-user.target")
      host.wait_for_unit("harness-cp.service")

        for _ in $(seq 1 60); do
          journalctl -u microvm@agent-01.service --no-pager \

        done
        exit 1
      """)

```

## 9. What this scenario explicitly does NOT exercise

Documented so reviewers don't expect more than what's in scope:

- Rollout state machine (`OpenRollout` / `DispatchHost` / `PromoteWave`). The reconciler is unit-tested; the harness doesn't need to re-test the pure function.
- Real agent binary (`crates/agent`). Will swap in when Stream C Milestone 1 Part 3 ships.
- Real CP binary (`crates/control-plane`). Same.
- Activation (`nixos-rebuild switch`). Phase 4.
- Compliance probes. Checkpoint 2.
- Rollback. Checkpoint 2.
- Attic closure verification (attic signature). Separate agent-path, folded in later.
- p256 signing. Exercised by Stream C unit tests; redundant here.
- Rotation (two keys in the trust file). A separate harness scenario later.

## 12. Decisions (locked for the implementation PR)

### 12.1 Fixture derivation location

**Decision: `tests/harness/fixtures/signed/default.nix`.** Scenario-local. Not under `crates/nixfleet-proto/tests/fixtures/`.

Rationale. The fixture is a Nix derivation (openssl + canonicalize binary); it does not belong under Rust test fixtures. Scenario-local layout means later Checkpoint 2 scenarios (magic rollback, stale refusal, tamper refusal) each get a sibling derivation that copies the pattern with one-input tweaks.

### 12.2 Keypair generation

**Decision: fixed seed.** The fixture derivation derives the ed25519 keypair from a hardcoded 32-byte seed constant.

Rationale. Nix build purity already catches non-determinism via output-hash stability; per-build keypairs add rebuild churn to CI for zero signal. The harness tests cross-stream byte-identity, not keypair entropy. Rotation scenarios later use a second fixed seed — deterministic variation by hand.

Concrete seed: `builtins.substring 0 32 (builtins.hashString "sha256" "nixfleet-harness-test-seed-2026")`. Changing the string forces a new keypair across every consumer of the fixture.

### 12.3 Signature algorithm for the first wire-up

**Decision: ed25519.** Not ecdsa-p256.

Rationale. Both algorithm paths through `verify_artifact` are already unit-tested in Stream C (`verify_p256_ok`, `verify_p256_rejects_high_s`). The harness's purpose is cross-stream wiring (mTLS + canonicalize + verify composition), not algorithm coverage. ed25519 gives the simplest fixture derivation (openssl or ed25519-dalek), no TPM simulation.

Follow-up scenario (not this PR): `fleet-harness-signed-rotation-cross-algo` — pairs ed25519 `current` + ecdsa-p256 `previous` in `trust.json` to exercise cross-algorithm rotation end-to-end.

### 12.4 `nixfleet-verify-artifact` CLI shape

**Decision: `--trust-file <path>`, not flat `--ciReleaseKey-*` args.**

Rationale. Mirrors the production binary shape — CP and agent both read trust from a file per `docs/trust-root-flow.md` §3.2. The harness's job is to exercise production shapes; flat args would create a CLI divergence that breaks when the thin wrapper is retired and the real agent takes over.

Final CLI surface: `nixfleet-verify-artifact --artifact <path> --signature <path> --trust-file <path> --now <rfc3339> --freshness-window-secs <n>`.

### 12.5 Ordering relative to `docs/trust-root-flow.md`

**Decision: concurrent work, sequenced merge.** `docs/trust-root-flow.md` (PR #25) merges before `nixfleet#29` (Stream C M1 P3), which merges before Phase 2 PR (a) [`nixfleet-verify-artifact` CLI + `TrustConfig` + signed-fixture derivation].

Rationale. The doc is docs-only, fast to review. Stream C can code against the doc's current text in parallel; they just hold their merge until the doc is blessed. Phase 2 PR (a) consumes the blessed `TrustConfig` shape — no divergence possible.

Ordering summary:
1. PR #25 (`docs/trust-root-flow.md`) — merge once §7 decisions are reviewed.
2. `nixfleet#29` (Stream C M1 P3 — v0.2 agent + CP + CLI skeletons) — merge once PR #25 is in main.
3. Phase 2 PR (a) — merge once `nixfleet#29` is in main.
4. Phase 2 PR (b) — merge last, consumes all of the above.
